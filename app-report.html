Managing Machines That Build Machines

These apps are purpose-built for the Grotap operations team to run, monitor, and scale the autonomous agent infrastructure. They are the control layer — the tools that let humans steer the fleet of AI agents that build the platform itself.

The Self-Building Platform

Grotap runs a fleet of 10 Hetzner agent servers (agent-01 through agent-10) executing software engineering tasks autonomously. A continuous dispatch loop assigns tasks from a queue to available servers, each running Claude Code in isolated git worktrees. The apps below give the team visibility into this machine, the ability to intervene when agents get stuck, and the tools to manage the AI workforce itself.

How These Apps Work Together

The operational flow starts with Agent Pipeline where tasks are submitted. The dispatch system routes them to available servers visible in Servers and Farm Dashboard. AI Agents shows real-time execution. When an agent gets stuck, Human Intervention fires a push notification and creates a hold record. Approvals handles any human-in-the-loop decisions agents request during execution.

Operational Flow

Agent Pipeline → Dispatch System → Servers / Farm Dashboard → AI Agents → Human Intervention / Approvals

Platform Framework & Architecture

Technology Stack

  • Frontend: React + Vite + TypeScript, deployed on Vercel (apps.grotap.com, agents.grotap.ai)
  • Backend: FastAPI (Python), deployed on Railway (api.grotap.com), 39+ routers
  • Agent Worker: TypeScript + LangGraph, deployed on Railway
  • Ingestion Worker: TypeScript + INNGEST, deployed on Railway
  • Mobile: Expo React Native (7 apps have mobile support)
  • Database: Neon Serverless Postgres — database-per-tenant isolation
  • Auth: WorkOS AuthKit (currently on staging — production migration pending)
  • Secrets: Doppler (single source of truth, never GitHub secrets)
  • Infrastructure: 10 Hetzner agent servers, Terraform-managed

Database-Per-Tenant Architecture

Every tenant gets its own Neon Postgres project, fully isolated. The control plane database (green-rice-76766370) stores the app catalog, brands, tenant registry, and cross-tenant metadata. Tenant-specific data (documents, RFID scans, forecasts, etc.) lives exclusively in the tenant's own database, provisioned dynamically by NeonProvider.

App Registry & Marketplace

Apps follow a lifecycle: review → approved → development → beta_testing → production_live. The /app-registry router exposes the full catalog with subscription status. Tenants can subscribe to apps, which grants features via WorkOS. The App Library page also includes a hardcoded quick-access module grid.

App categories: Finance, Legal, HR, Operations, Platform, Infrastructure, Internal.

Key Architectural Rules

  • Agents = TypeScript/JS only. Python = FastAPI backend only.
  • All 3rd-party calls go through vendor wrapper classes (providers/).
  • No vector embeddings — PageIndex reasoning-based retrieval only.
  • All secrets in Doppler. The only GitHub secret is DOPPLER_SERVICE_TOKEN.

Multi-Brand Strategy

Two Brands, One Platform

The platform runs two distinct brand experiences from the same codebase and infrastructure:

🌐 grotap-apps — apps.grotap.com

The primary tenant-facing brand. Hosts all platform core apps (Dashboard, Billing, Analytics, Settings, etc.), operational tools (RFID Pipe, Forecast Tool), support tools, and the admin panel. This is what customers see and use.

18 apps assigned — everything except the agent-specific tools.

🤖 grotap-agents — agents.grotap.ai

The AI operations brand. Purpose-built for managing the agent fleet. Hosts Agent Pipeline, AI Agents, Farm Dashboard, Servers, plus Brand Management and App Manager for cross-brand administration.

6 apps assigned — focused exclusively on the AI agent infrastructure.

Brand Architecture

Brands are stored in the brands table (control plane). Each brand has a slug, display name, domain, and logo. The brand_app_assignments junction table maps which apps appear on which brand. The brand_users table assigns per-brand roles (admin, member, viewer) — replacing the old @grotap.com email check for admin access.

Brand Management and App Manager are assigned to both brands, allowing platform admins to manage the multi-brand ecosystem from either interface. App Manager is gated to grotap-agents brand admins only for write operations.
